# Koda V2 — off-hardware simulator

Test the control system without the robot. The simulator runs the **real ESP32 control
core** (the firmware C++ compiled to WebAssembly) against a reduced-physics world, in the
browser and headless under node. There is **no second copy** of the control logic — the
same `.cpp` files that build for the ESP32 build here, so the sim can never test stale
logic.

```
 ┌──────────── browser / node (JS) ────────────┐      ┌──────────────────────────┐
 │  world.js  reduced physics:                  │      │  koda-core.wasm          │
 │   servo/foot targets → leg placement         │ ──▶  │  = firmware control core │
 │   foot vs terrain → spring compression →     │      │  IK · gait · balance ·   │
 │   per-foot force (the hall reading)          │ ◀──  │  ground-contact · robot  │
 │  render.js side + top views · main.js drive  │      └──────────────────────────┘
 └──────────────────────────────────────────────┘            ▲ same source as
                                                              esp32-master (PlatformIO)
```

## One-time setup — Emscripten

The WASM build needs Emscripten (it ships its own clang, so no system C++ compiler is
required):

```bash
git clone https://github.com/emscripten-core/emsdk.git ~/emsdk
cd ~/emsdk && ./emsdk install latest && ./emsdk activate latest
```

`build.sh` sources `~/emsdk/emsdk_env.sh` automatically.

## Build + test (headless)

```bash
cd "V2 Code/sim"
npm run build        # bash wasm/build.sh  → web/koda-core.mjs (+ .wasm)
npm test             # node core.test.mjs && node world.test.mjs
```

The tests exercise the actual firmware logic (via WASM), so they double as regression
tests for the control system — runnable in CI.

## Run the interactive sim

WASM must be fetched over HTTP (not `file://`), so serve the folder:

```bash
cd "V2 Code/sim/web"
python -m http.server 8000      # or: npx serve
# open http://localhost:8000
```

Press **B** to stand, then drive. Keys:

| Key | Action | Key | Action |
| --- | ------ | --- | ------ |
| `B` | stand / sleep | `W`/`S` | forward / back |
| `X` | sit | `A`/`D` | strafe |
| `G` | cycle gait | `Q`/`E` | turn |
| `Space` | play / pause | `R`/`F` | body height |

Use the **Speed** dropdown for slow-motion (down to 0.05×) to study a manoeuvre — the
control + physics run at a fixed 50 Hz tick regardless of speed, so 0.1× is true slow
motion, not a different simulation.

**Scenarios** (Terrain dropdown):
- **flat** — basic walking; watch the gait, the **5-bar linkage** of each leg, and the
  per-foot force bars. The **walk start is a half-step**: the gait begins mid-swing so the
  first move is a half-step *forward* from the standstill pose — no leg slides backward
  through stance against the floor (no foot dragging). The **stop** winds the stride and
  lift down to a planted neutral stance over a cycle (through a STOPPING state). The sim's
  body is driven by its **stance feet** (a planted foot stays put in the world, as on the
  real robot), so dragging shows up faithfully if the gait ever causes it.
- **slope (incline)** — press B to stand: the integral balance shifts the body until the
  **CoG sits over the support centroid** (top view: yellow dot over grey). Then drive: with
  **Auto slope** on, the robot detects the slope from the contact-event pattern, auto-stops
  (passing through STOPPING → MEASURING), **measures the slope from the springs**, and
  resumes walking with the CoG held centred by feed-forward. The readout shows the live and
  measured slope angle and the terrain class. Tick "Sim IMU" to preview torso-levelling.
- **step up** — walk forward into the platform: a swinging foot meeting the step early
  flags **early ground contact** (foot turns red, the leg plants).
- **step down** — the ground drops away: a stance foot that finds no ground flags **late
  ground contact** (foot turns orange) and the **leg reaches down to find the floor** so the
  body keeps its support instead of dropping. If the drop is beyond the leg's reach the foot
  goes magenta (out of range / no IK) — the controller can't support there.
- **rolling bumps** — uneven terrain (exercises both early and late contact).

The **side view** shows each leg's full **5-bar linkage** (two knee servos → thighs →
knees → calf springs → foot), the sagittal pose, pitch and per-foot force bars. The
**top view** shows the support polygon with the **CoG (yellow) over the support centroid
(grey)** and a line between them — which is what makes the incline balancing legible. The
readout includes the live `CoG−centroid` offset in mm.

**Step geometry** (panel) is tunable live and feeds straight into the real control core:
- **Stance height** — neutral leg drop (body height).
- **Stance width** — lateral foot track.
- **Step height** — swing-arc lift.
- **Step length** — forward stride.

If a commanded foot target can't be reached (you crank the stance past the leg's reach,
or a manoeuvre drives a foot out of the workspace), the **IK has no solution**: that leg
turns **magenta** in both views, the readout shows `⚠ OUT OF RANGE`, and an event is
logged. This is the controller's real per-leg IK result (it holds the last good pose for
an unreachable leg), so you can see exactly when your geometry or commands exceed the
mechanism.

## Debugging & logging

### Debuggers — three levels

**1. JS in the browser (the physics + how the sim drives the core).** Open DevTools → Sources,
set breakpoints in `world.js` / `main.js` / `render.js`. They're plain ES modules, no
transpile, so what you see is what runs. Pause and inspect `world.body`, `world.forces`,
the values going into `core.step`, etc.

**2. JS under node (headless, great for the physics).**
```bash
node --inspect-brk test/world.test.mjs     # then open chrome://inspect, or attach VS Code
```
A VS Code launch config (`"type": "node"`, program = the test) lets you breakpoint in
`world.js` and the tests directly.

**3. The actual C++ control core, at source level.** Build with debug info and step through
`robot.cpp` / `balance.cpp` / `gait.cpp` in Chrome:
```bash
npm run build -- --debug          # or: bash wasm/build.sh --debug   (embeds DWARF)
```
Install Chrome's **“C/C++ DevTools Support (DWARF)”** extension, serve `web/`, open it, and in
DevTools → Sources you'll see the `.cpp` files. Set breakpoints in the real firmware logic,
step, and inspect C++ variables — the same code that runs on the ESP32. Rebuild without
`--debug` for the small/fast release wasm when you're done.

> Native `gdb`/`lldb` debugging would need a host C++ toolchain (we use Emscripten precisely
> to avoid that). The DWARF-in-Chrome path above gives you source-level C++ debugging without it.

### Logging & traces

The page records every tick. In the **Logging** panel:
- **record trace** — capture each tick (command, body pose, per-foot force, contact/early
  flags, all 12 servo angles).
- **console** — also print notable events (state changes, early-contact, planted/lifted) to
  the browser console.
- **⬇ CSV / ⬇ JSON** — download the trace; the live event feed shows the last ~200 events.

**Headless trace → CSV** (debug offline / plot it):
```bash
npm run trace -- slope 600 12 > slope.csv     # node tools/trace.mjs slope 600 12
npm run trace -- step  800    > step.csv
```
Columns are documented in `web/logger.js` (`flatten`). Each run stands, then walks; load it
into a spreadsheet, pandas, or any plotter to see exactly what the controller did — e.g.
plot `f_FL,f_FR,f_RR,f_RL` to watch the balance loop redistribute load on a slope, or
`early_FL…` against `phase` to see early-contact firing on the step.

The same `Recorder` (`web/logger.js`) works in your own node scripts:
```js
import { Recorder } from './web/logger.js';
const rec = new Recorder();
rec.push(world.state(), dt);   // each tick
fs.writeFileSync('trace.csv', rec.toCSV());
```

## Automatic slope handling (the control model)

The control philosophy split: **static = full spring sensing, walking = contact events only.**
- **Standing:** the springs give a trustworthy continuous force reading. `SlopeEstimator`
  reads the body pitch/roll from the force-weighted centre of pressure, and the live integral
  balance centres the CoG.
- **Walking:** hall sampling is too slow to balance on live force, so balance is **feed-forward**
  from the last measured slope; the sensors are used only for **early/late contact events**.
- **TerrainMonitor** watches the consistency of those events: a coherent directional bias = a
  **slope** (triggers the automatic *stop → MEASURING → adjust → resume* cycle); scattered =
  **uneven** ground (left to the per-step early/late handling).

### How well it works (sim evaluation)
- **Fore/aft slopes (±12–16°):** auto-detected, measured to ~10 % of true, CoG re-centred to
  ~7–12 mm (steeper slopes clamp at the 45 mm shift limit). Up- and down-hill both stable.
- **Side slopes (±12°):** detected and laterally centred to ~10 mm; measured roll ~10–15°
  for 12° true. (The lateral support base is narrow, so the body settles slightly less than
  fully parallel — the sim's roll settle gain is scaled by the geometry to compensate.)
- **Sensitivity floor:** gentle slopes (≲10°) produce too few consistent contact events to
  auto-trigger — inherent to event sensing. Finer contact detection (see
  [SPRING_SIZING.md](../SPRING_SIZING.md)) or an IMU would extend it.
- **Uneven ground** occasionally trips the measure cycle; it's benign (reads ~flat, applies
  ~no bias) and rate-limited by a cooldown.
- **Re-triggering** on a known slope is suppressed (≈1 measure cycle / 40 s): after the COG
  correction the residual events scatter and classify as uneven. (A gait *slope-follow* knob
  exists but is **off** by default — pre-tilting the feet double-tilts against the body that
  already settles parallel to the slope, which the sim sweep confirmed makes things worse.)

## Spring advisor

The **Spring advisor** panel runs the [spring-sizing maths](../SPRING_SIZING.md) live on the
current settings: it derives the usable spring-rate window (soft enough to resolve slopes,
stiff enough to detect contact), recommends a rate + travel, and flags when the current
modelled spring sits inside it.

For the full tool with every input exposed (sensor calibration, ADC noise, preload, travel,
slope range…) plus a rate-window bar and derived-quantity readout, open the dedicated
**[web/spring.html](web/spring.html)** page — it's pure JS (no WASM needed). The same maths is
also a CLI: `node tools/spring_sizing.mjs`.

## How it stays a single source of truth

`wasm/build.sh` compiles exactly the firmware control modules:

```
kinematics/inverse_kinematics.cpp  gait/gait.cpp  control/leg.cpp
control/robot.cpp  control/balance.cpp  control/ground_contact.cpp  (+ bindings.cpp)
```

plus the pure headers (`math/*`, `sensors/hall_model.h`, `sensors/imu.h`, `config.h`).
Those are the same files PlatformIO builds for the ESP32. Change the controller once; the
firmware and the sim both get it. (This is why the old Python cross-check mirrors are no
longer needed — `core.test.mjs` checks the real thing.)
